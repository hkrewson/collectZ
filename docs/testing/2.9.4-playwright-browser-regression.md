# 2.9.4 Playwright Browser Regression Foundations

This milestone adds a first-class Playwright harness so collectZ can verify key browser-visible workflows in addition to backend and API smoke coverage.

## Tracking and Packaging Boundary

- The root Playwright manifest (`package.json` + `package-lock.json`) is tracked in git as repo test infrastructure.
- The Playwright harness is part of CI and release confidence, not part of the shipped runtime application.
- Production app images do not install the root Playwright toolchain:
  - backend runtime image is built from `backend/`
  - frontend runtime image is built from `frontend/`
- Browser regressions run against the live compose stack from the CI runner or a dedicated test environment alongside the app containers.
- Saved Playwright auth state is kept under `tmp/playwright-auth/`, outside the uploaded failure-artifact tree.

## Local Run

1. Ensure the local Docker stack is running:

```bash
PLAYWRIGHT_E2E_BYPASS_TOKEN=collectz-playwright APP_VERSION=2.9.2 docker compose --env-file .env up -d --build backend frontend
```

2. Install the root browser-test dependencies:

```bash
npm install
npx playwright install chromium
```

3. Run the browser suite:

```bash
PLAYWRIGHT_E2E_BYPASS_TOKEN=collectz-playwright npm run test:browser
```

4. Optional interactive runs:

```bash
npm run test:browser:headed
npm run test:browser:ui
```

5. Generate intentional support/docs screenshots:

```bash
PLAYWRIGHT_E2E_BYPASS_TOKEN=collectz-playwright npm run test:browser:capture
```

## Runtime Expectations

- Base URL defaults to `http://127.0.0.1:3000`
- Override with `PLAYWRIGHT_BASE_URL` if needed
- When `PLAYWRIGHT_E2E_BYPASS_TOKEN` is set, Playwright sends `x-playwright-e2e-bypass` automatically so browser regression runs skip app-layer rate limiting without changing normal runtime behavior
- Browser-driven login and in-app mutation flows also persist a dedicated `playwright_e2e_bypass` cookie when the token is set so real page interactions stay deterministic across reruns.
- The authenticated setup project bootstraps an admin browser session automatically:
  - it first tries the configured admin credentials
  - then falls back to a direct test admin bootstrap if login/register cannot produce a usable session

## Covered Foundation Flows

- browser login and logout
- authenticated admin shell load
- docs surface availability when authenticated admin + debug gating are satisfied
- `All Spaces` drawer tab switching
- support-session start/end banner behavior
- Integrations tab switching and save feedback
- Integrations critical flows:
  - Barcode settings save and persist after reload
  - Games settings save and persist after reload
  - Metrics Export toggle persists after reload
  - integrations section-tab strip snapshot stays stable
  - route-driven admin integration deep links honor supported sections such as `metrics` and `logs`
- end-user Help Center support interactions:
  - create request,
  - reply,
  - close/reopen,
  - history view,
  - releases detail expansion
- Help Admin support workspace interactions:
  - select request from queue,
  - reply in the selected-case workspace,
  - save triage updates,
  - inspect request history,
  - close and confirm completed-queue visibility
- approved support-request session flow:
  - requester-side approval seeded through authenticated helpers,
  - support-admin session start from Help Admin,
  - support-session banner evidence,
  - support-library switching while tenant support is active,
  - temporary `My Space` availability only while the approved session is active
- import/manual-entry flow:
  - end-user barcode lookup in `Import Media`,
  - deterministic stubbed UPC lookup response for browser stability,
  - real add-to-library POST through the live app,
  - library search confirms the imported title is visible after add,
  - authenticated API cleanup removes seeded browser-imported media after the test
- CSV import flow:
  - end-user generic CSV upload through the real `Import Media` browser UI,
  - browser verifies the queued import response and visible queued-job message,
  - authenticated helper polling waits for the exact queued import job to finish,
  - library search confirms the CSV-imported title is visible after the job succeeds,
  - authenticated cleanup removes seeded browser-imported media and the temporary CSV file after the test
- Events and Collectibles flow:
  - end-user event creation in the Events surface,
  - end-user collectible creation in the Collectibles surface,
  - collectible links to a created event through the real browser form,
  - collectibles detail view confirms the linked event relationship,
  - authenticated cleanup removes seeded browser-created events and collectibles after the test
- negative-path browser boundaries:
  - support admins do not see admin/global/import shell surfaces by default,
  - support admins hitting tenant-library routes are redirected back into `Help Admin`,
  - support admins hitting admin routes are also redirected back into `Help Admin`,
  - standard users hitting admin routes see the admin-permission restriction view
- edition-boundary browser coverage:
  - `homelab` Help mounts `Guidance` and `Releases` only,
  - `homelab` never mounts `Metrics`, `Support`, or `Help Admin`,
  - direct `support-inbox` navigation in `homelab` redirects back to `Help`,
  - homelab admins do not get `Global`, `All Spaces`, `All Members`, `Activity`, or `My Space` shell affordances,
  - direct homelab navigation to platform control-plane routes such as `admin-spaces`, `admin-users`, `admin-activity`, and `space-manage` redirects to a homelab-safe admin landing surface
- edition-safe shared workflow coverage:
  - `homelab` end users can still sign in, browse the library, import media, and update their profile,
  - `homelab` admins can still load valid shared settings and integrations surfaces,
  - shared homelab browser flows restore seeded integration/settings state after save coverage so edition verification stays reusable
- edition-boundary backend mounting verification:
  - `homelab` still exposes shared Help release data through `/api/support/releases`,
  - `homelab` does not mount platform-only support request APIs, docs, metrics, spaces, or platform admin control-plane routes,
  - `homelab` shared auth/scope/library responses keep `active_library_id` and library lists available while returning `active_space_id: null` and `spaces: []` so the single internal default space stays an implementation detail,
  - `homelab` registration remains invite-free after the first admin so the edition no longer depends on the platform tenant-invite model for local account creation
  - use the backend smoke gate `npm run test:homelab-edition-boundary` against a `homelab` stack to verify the mounted shared surfaces and unmounted platform-only APIs even when local browser execution is unavailable
  - `platform` still mounts those APIs normally

## Failure Artifacts

Playwright keeps the following on failure under `artifacts/playwright/`:

- traces
- screenshots
- videos
- HTML report

It does not store reusable auth state or saved browser credentials in that uploaded artifact directory.

## CI Pattern

The intended CI sequence mirrors local Docker-first execution:

```bash
PLAYWRIGHT_E2E_BYPASS_TOKEN=collectz-playwright APP_VERSION=2.9.2 docker compose --env-file .env up -d --build backend frontend
npm ci
npx playwright install chromium
PLAYWRIGHT_E2E_BYPASS_TOKEN=collectz-playwright npm run test:browser
```

Playwright complements the existing backend/API/release gates rather than replacing them.

In GitHub Actions, browser regression should remain a dedicated gate that:

1. installs the root Playwright dependencies,
2. installs Chromium,
3. boots the compose stack with `PLAYWRIGHT_E2E_BYPASS_TOKEN` enabled,
   - CI should generate a random token per run rather than using a fixed literal value,
4. runs `npm run test:browser`,
5. uploads `artifacts/playwright/` when failures occur.

Intentional documentation/support screenshot generation should stay separate from the blocking browser-regression gate.

- Use the dedicated manual workflow:
  - `.github/workflows/browser-captures.yml`
- That workflow:
  1. boots the compose stack with a per-run random bypass token,
  2. runs `npm run test:browser:capture`,
  3. uploads `artifacts/playwright/captures/` as a reusable screenshot artifact.

## Next Ergonomics Direction

The intended next steps for the Playwright layer are:

1. expand import/manual-entry and other critical library workflows now that support/admin/integrations flows are covered,
2. expand into CSV/manual-entry or adjacent critical library workflows without pulling in camera-specific flakiness yet,
3. expand into the next adjacent library workflow or the highest-risk remaining critical path without pulling in camera-specific flakiness,
4. add light shared page/workspace helpers where repeated browser flows start to get noisy,
5. keep named captures and focused visual snapshots stable enough that support/docs screenshots remain reusable over time.
