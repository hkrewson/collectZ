# 2.9.4 Playwright Browser Regression Foundations

This milestone adds a first-class Playwright harness so collectZ can verify key browser-visible workflows in addition to backend and API smoke coverage.

## Tracking and Packaging Boundary

- The root Playwright manifest (`package.json` + `package-lock.json`) is tracked in git as repo test infrastructure.
- The Playwright harness is part of CI and release confidence, not part of the shipped runtime application.
- Production app images do not install the root Playwright toolchain:
  - backend runtime image is built from `backend/`
  - frontend runtime image is built from `frontend/`
- Browser regressions run against the live compose stack from the CI runner or a dedicated test environment alongside the app containers.
- Saved Playwright auth state is kept under `tmp/playwright-auth/`, outside the uploaded failure-artifact tree.

## Local Run

1. Ensure the local Docker stack is running:

```bash
PLAYWRIGHT_E2E_BYPASS_TOKEN=collectz-playwright APP_VERSION=2.9.2 docker compose --env-file .env up -d --build backend frontend
```

2. Install the root browser-test dependencies:

```bash
npm install
npx playwright install chromium
```

3. Run the browser suite:

```bash
PLAYWRIGHT_E2E_BYPASS_TOKEN=collectz-playwright npm run test:browser
```

4. Optional interactive runs:

```bash
npm run test:browser:headed
npm run test:browser:ui
```

5. Generate intentional support/docs screenshots:

```bash
PLAYWRIGHT_E2E_BYPASS_TOKEN=collectz-playwright npm run test:browser:capture
```

## Runtime Expectations

- Base URL defaults to `http://127.0.0.1:3000`
- Override with `PLAYWRIGHT_BASE_URL` if needed
- When `PLAYWRIGHT_E2E_BYPASS_TOKEN` is set, Playwright sends `x-playwright-e2e-bypass` automatically so browser regression runs skip app-layer rate limiting without changing normal runtime behavior
- The authenticated setup project bootstraps an admin browser session automatically:
  - it first tries the configured admin credentials
  - then falls back to a direct test admin bootstrap if login/register cannot produce a usable session

## Covered Foundation Flows

- browser login and logout
- authenticated admin shell load
- docs surface availability when authenticated admin + debug gating are satisfied
- `All Spaces` drawer tab switching
- support-session start/end banner behavior
- Integrations tab switching and save feedback

## Failure Artifacts

Playwright keeps the following on failure under `artifacts/playwright/`:

- traces
- screenshots
- videos
- HTML report

It does not store reusable auth state or saved browser credentials in that uploaded artifact directory.

## Intentional Capture Output

Capture-mode screenshots are written to:

- `artifacts/playwright/captures/`

These are distinct from failure artifacts and are intended for:

- support-reference screenshots
- help/documentation drafting
- release or UX evidence when a stable visual snapshot is useful

## CI Pattern

The intended CI sequence mirrors local Docker-first execution:

```bash
PLAYWRIGHT_E2E_BYPASS_TOKEN=collectz-playwright APP_VERSION=2.9.2 docker compose --env-file .env up -d --build backend frontend
npm ci
npx playwright install chromium
PLAYWRIGHT_E2E_BYPASS_TOKEN=collectz-playwright npm run test:browser
```

Playwright complements the existing backend/API/release gates rather than replacing them.

In GitHub Actions, browser regression should remain a dedicated gate that:

1. installs the root Playwright dependencies,
2. installs Chromium,
3. boots the compose stack with `PLAYWRIGHT_E2E_BYPASS_TOKEN` enabled,
   - CI should generate a random token per run rather than using a fixed literal value,
4. runs `npm run test:browser`,
5. uploads `artifacts/playwright/` when failures occur.

## Next Ergonomics Direction

The intended next steps for the Playwright layer are:

1. screenshot-friendly capture mode for support/help documentation,
2. small seeded visual fixtures for stable queue/help/admin states,
3. incremental expansion across the most brittle user-visible flows before broadening coverage further.
