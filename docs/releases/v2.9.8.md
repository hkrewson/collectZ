## Summary
This release ships two connected slices of hardening work. First, the app’s add/edit drawers, Help surfaces, and tabbed workflows were normalized onto calmer shared UI primitives so the product reads more consistently and accessibly across media, events, collectibles, help, and admin flows. Second, the runtime observability story was hardened for longer-lived operator use with clearer diagnostics, supported collector-path smokes, persistence rehearsals, and a repeatable observability release-evidence artifact that proves logs and metrics stay recoverable without turning collectors into a hidden runtime dependency.

## What Changed
### Shared tabbed-editor and drawer consistency
- Normalized the media, event, collectible, help, help-admin, admin-integrations, and all-spaces surfaces onto shared tab/disclosure primitives.
- Added complete tab semantics for true tabbed surfaces and kept button semantics for filter/mode rows that only borrow the same visual treatment.
- Reduced AI-style card/pill overuse in Help, Help Admin, and event schedule flows in favor of flatter rows, calmer list sections, and simpler expandable guidance items.
- Tightened add/edit drawer composition across media, events, and collectibles with a consistent 40rem shell, title-first field hierarchy, and synchronized add/edit field placement.

### Help and guidance polish
- Converted Help guidance into one-open-at-a-time inline disclosure rows so the help surface scales without becoming a long wall of open content.
- Flattened Help / Help Admin tab rails and guidance/release layouts into quieter list-based sections instead of decorative card grids and KPI tiles.
- Left a documented future accessibility follow-up for URL-synced tabs and expanded guidance state where deep-linking provides real value.

### Observability runtime hardening
- Added operator-facing runtime diagnostics for external logs and metrics.
- Surfaced drift clues such as exporter/backend mismatch, wrong collector host or port, weak scrape-token posture, reverse-proxy exposure expectations, and mixed stdout versus structured-export caveats.
- Added clearer non-blocking guidance so exporter failures remain additive rather than becoming a hidden availability dependency.

### Supported collector-path verification
- Added repeatable collector-path smokes for Graylog / GELF, Loki / Promtail, and syslog.
- Added a repeatable non-blocking failure rehearsal that proves audit activity and `/api/health` remain healthy even when the configured collector is intentionally unreachable.
- Added a structured-only Loki/Promtail option for operators who want cleaner structured export ingestion without the full mixed stdout stream.

### Persistence, backup, and recreate confidence
- Added repeatable persistence/recreate rehearsals for Prometheus + Grafana, Graylog + MongoDB + OpenSearch, and Loki + Promtail positions/state.
- Tightened docs to distinguish local/dev example stacks from longer-lived operator guidance.
- Clarified the difference between normal stack recreation and destructive `down -v` reset paths.

### Release-shaped observability evidence
- Added a release-facing observability evidence runner: `npm --prefix backend run test:observability-evidence`.
- The generated artifact now captures collector-path evidence, non-blocking failure evidence, persistence/recreate evidence, and backend restore plus final stack health.
- The current artifact path is `artifacts/observability-evidence/observability-release-evidence.json`.

## Breaking Changes
- No API-breaking schema or runtime contract change is introduced in `2.9.8`.
- Admin/operator surfaces now present observability state and guidance more explicitly, but existing export backends remain opt-in and non-blocking.

## Environment and Config Changes
- No new mandatory runtime env vars are required for normal application startup.
- Existing observability env remains supported, including `LOG_EXPORT_BACKEND`, `LOG_EXPORT_HOST`, `LOG_EXPORT_PORT`, `LOG_EXPORT_DEBUG`, `METRICS_SCRAPE_TOKEN`, and `DEBUG`.
- Graylog example operators still need `GRAYLOG_PASSWORD_SECRET` and `GRAYLOG_ROOT_PASSWORD_SHA2` for their own long-lived stack configuration outside the temporary release-evidence automation path.

## Migration and Data Impact
- No new schema migration is introduced as part of `2.9.8`.
- Persistence rehearsals validate example observability stack survival across normal Docker recreates, not destructive volume deletion.
