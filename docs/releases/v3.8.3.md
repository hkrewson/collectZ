# collectZ v3.8.3

## Version and date

- Version: `3.8.3`
- Date: `2026-05-18`

## Milestone target and status

- Milestone: `3.8.3 — Capture Offline Queue Idempotency`
- Status: `implemented`

## Summary

This release makes Capture Inbox intake safer for mobile/scanner offline queues by letting clients resend queued captures without creating duplicate rows.

## What changed

- Added optional `client_capture_id` and `client_source` handling for quick-capture records.
- Reusing the same `client_capture_id` in the same workspace/library scope now returns and updates the existing capture item instead of inserting another row.
- Added idempotent replay behavior for photo uploads, including returning the existing image path when the original upload already succeeded.
- Exposed normalized `client_capture_id`, `client_source`, and idempotency readback in capture API responses.
- Documented the idempotent capture create and upload behavior in OpenAPI.

## Breaking changes

- None.

## Environment/config changes

- No new environment variables.

## Migration and data impact

- No new database migration.
- Client retry keys are stored in existing `capture_items.source_context` metadata.

## Security triage

- Capture create and upload routes remain authenticated and scope-enforced.
- Personal access token scope mapping continues to use `media:write` for capture mutations.
- Idempotency lookup is scoped to the active workspace/library so client retry keys do not cross tenant or library boundaries.

## Security vulnerability triage

- High findings: none identified in local release checks for this slice.
- Owner: collectZ maintainers.
- Target remediation milestone: before `3.8.3` closeout if any high findings appear.

## Deployment and verification

- Verify `/api/health` reports frontend/backend/build `3.8.3`.
- Verify Help > Releases includes `3.8.3`.
- Verify `POST /api/capture-items` returns the same item with `idempotent: true` when `client_capture_id` is replayed.
- Verify `POST /api/capture-items/upload-image` returns the same photo capture and image path when the same `client_capture_id` is replayed.

## Rollback guidance

- Revert the idempotency route/contract changes and restore `3.8.2` version metadata and release feed if scanner retry behavior causes regressions.
- No schema rollback is required for this slice.

## Known issues and follow-up

- This adds retry safety, not full offline queue management inside the web app.
- A future slice can add richer conflict/review handling for queued captures that are resent with meaningfully different metadata.
