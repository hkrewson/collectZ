# v2.6.0

## Version and date
- Version: `2.6.0`
- Date: `2026-03-13`

## Milestone target and status
- Target milestone: `2.6.0 — Observability Platform (Metrics + Alerting)`
- Status: `Completed with a deferred baseline-tuning follow-up`

## Summary
This release adds a real observability foundation for collectZ: structured metrics, a gated metrics endpoint, OpenAPI as a maintained API contract, an admin-only docs surface, a runnable Prometheus/Grafana starter stack, and the first operator-facing dashboards, alert rules, and triage guidance.

## What changed
### Backend
- Added Prometheus-style metrics export covering:
  - API request volume and failures
  - request latency buckets
  - auth outcomes
  - import lifecycle outcomes
  - import enrichment outcomes
  - provider request outcomes
  - admin mutation outcomes
  - current sync-job status counts
- Added `/api/metrics` as an admin-gated observability surface with:
  - `DEBUG>=1`
  - `metrics_enabled=true`
  - support for `METRICS_SCRAPE_TOKEN` for trusted internal scrapers
- Added `/api/docs` and `/api/docs/openapi.json` as admin-only API docs surfaces gated by:
  - `DEBUG>=1`
  - `api_docs_enabled=true`
- Added request/auth/import/provider instrumentation for:
  - Plex
  - TMDB
  - Metron
  - CSV and Delicious import paths

### API contract and validation
- Added `backend/openapi/openapi.yaml` as the maintained contract baseline.
- Added OpenAPI validation script and CI enforcement.
- Expanded contract coverage to include:
  - auth session + token flows
  - service-account keys
  - admin invites
  - media import launch + async job status/result
  - docs and metrics surfaces

### Observability and ops
- Added operator documentation:
  - `docs/wiki/29-Metrics-and-Alerts.md`
  - `docs/wiki/30-Observability-Triage-Runbook.md`
  - `docs/wiki/31-Observability-Dashboard-Spec.md`
  - `docs/wiki/32-Alert-Rules-Spec.md`
  - `docs/wiki/33-Prometheus-and-Grafana-Integration-Guide.md`
  - `docs/wiki/34-Observability-Baseline-Tuning-Log.md`
- Added starter monitoring artifacts:
  - `ops/monitoring/docker-compose.monitoring.yml`
  - Prometheus scrape config
  - Grafana datasource provisioning
  - provisioned dashboard JSON
- Added alert rules for:
  - API failures and latency
  - auth failure spikes
  - import failures and queue pressure
  - provider-side request issues
  - Delicious no-match ratio warnings

### Dashboard highlights
- Added import outcomes and enrichment panels tuned for low-frequency background work using `increase(...[$__range])`.
- Added provider request and provider error panels to distinguish upstream issues from app logic issues.
- Added import quality ratio panels, starting with Delicious no-match ratio and a tracked-ratios row for future provider-specific quality signals.

## Breaking changes
- None intended for normal app consumers.
- `/api/docs` and `/api/metrics` are intentionally unavailable unless their admin/debug/feature-flag gates are satisfied.

## Environment/config changes
- Added/used:
  - `FEATURE_FLAG_API_DOCS_ENABLED`
  - `FEATURE_FLAG_METRICS_ENABLED`
  - `METRICS_SCRAPE_TOKEN`

## Migration and data impact
- No new schema migration was required for this milestone.
- Data impact is additive at runtime only:
  - new metrics series
  - new monitoring docs/artifacts
- No destructive data migration required.

## Deployment and verification
1. Rebuild and start:
   - `docker compose --env-file .env up -d --build backend frontend`
2. Verify reported app version is `2.6.0`.
3. Verify OpenAPI baseline locally:
   - `npm --prefix backend run test:openapi`
4. Verify metrics/docs gates:
   - `/api/docs` returns `404` unless admin + `DEBUG>=1` + `api_docs_enabled=true`
   - `/api/metrics` returns `404` unless admin or `METRICS_SCRAPE_TOKEN` + `DEBUG>=1` + `metrics_enabled=true`
5. Start the starter monitoring stack:
   - `docker compose -f ops/monitoring/docker-compose.monitoring.yml up -d`
6. Verify:
   - Prometheus target is `UP`
   - Grafana shows the provisioned `collectZ Overview` dashboard
   - import/auth/admin/provider panels populate during normal use

## Rollback guidance
- Roll back application images if needed.
- Prometheus/Grafana data can remain forward; monitoring artifacts are additive.
- If metrics scraping causes operational issues, disable the metrics surface by removing the flag/token enablement and stop the monitoring stack.

## Known issues and follow-up
- Current alert thresholds are intentionally first-pass and should be revisited after more real import/provider traffic is observed.
- The starter monitoring stack is intended for local or protected internal use and does not yet include production-grade alert delivery wiring.
- Provider-specific quality ratios beyond Delicious should be added only after baseline evidence shows they are useful.
