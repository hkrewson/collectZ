## Summary
This release moves External Logs from an env-only operator setup toward a real admin-managed control plane. Admin can now configure the common external log endpoint fields inside `Admin -> Integrations -> External Logs`, validate the running target, review the latest validation result, and understand whether the current runtime is using saved settings, env defaults, or an immutable env override. The existing fail-safe behavior remains intact: external export stays additive, collector failure stays non-blocking, and the primary DB-backed audit path remains durable.

## What changed
### External Logs control plane
- Added admin-managed storage for external log backend / transport, collector host, collector port, service label, host label, and debug mode.
- Added read-only runtime override behavior through `LOG_EXPORT_SETTINGS_READ_ONLY=true` so locked-down deployments can keep env authoritative while still exposing state in the UI.
- Kept supported common-case targets explicit in the UI:
  - `gelf_udp`
  - `gelf_tcp`
  - `stdout_json`
  - `syslog_udp`
  - `syslog_tcp`

### Validation and diagnostics
- Added a `Validate` action in `Admin -> Integrations -> External Logs` that emits a deterministic audit/export event against the effective or candidate endpoint.
- Persisted the last validation result, including status, endpoint, and timestamp, so operators can see the last known proof without rerunning it immediately.
- Clarified validation semantics:
  - TCP backends pass when connect/write succeeds
  - `stdout_json` passes when the backend emits the structured event successfully
  - UDP backends report successful local send as a warning-level proof because downstream receipt is not acknowledged

### Runtime and audit integration
- Wired the control-plane fields into the runtime exporter resolution path instead of leaving the UI as a cosmetic overlay.
- Updated audit export event construction so saved service/host labels are part of the resolved runtime path used by structured-log export.
- Exposed runtime config-source diagnostics so the UI can distinguish:
  - saved admin settings
  - runtime env defaults
  - runtime env override / read-only mode

### External Logs UI normalization
- Flattened the `External Logs` panel so it reads like the rest of Admin instead of a stack of special-case cards.
- Moved `Debug Mode` to the main integration settings flow directly under the integration toggle.
- Collapsed `Last Validation` and `Runtime Checks` into quieter disclosure rows while keeping the primary configuration fields inline and editable.

### Operator docs and runbooks
- Updated structured-log export docs and observability runbooks to explain the new control-plane boundary, env fallback/read-only behavior, and validation expectations.
- Kept lower-level payload sizing (`LOG_EXPORT_MAX_DETAIL_BYTES`) documented as an env-only operator control rather than expanding the UI into an advanced tuning surface.

## Breaking changes
- No API-breaking media/auth/library contract changes are introduced in `2.9.9`.
- Deployments that rely entirely on env-based logging config continue to work; the UI now reflects and, unless locked down, can manage the common-case endpoint fields.

## Environment/config changes
- New optional runtime behavior:
  - `LOG_EXPORT_SETTINGS_READ_ONLY=true` keeps the External Logs control plane visible but immutable, with runtime env remaining authoritative.
- Existing logging env remains supported:
  - `LOG_EXPORT_BACKEND`
  - `LOG_EXPORT_HOST`
  - `LOG_EXPORT_PORT`
  - `LOG_EXPORT_HOST_LABEL`
  - `LOG_EXPORT_SERVICE`
  - `LOG_EXPORT_DEBUG`
  - `LOG_EXPORT_MAX_DETAIL_BYTES`
- `LOG_EXPORT_MAX_DETAIL_BYTES` remains env-only in `2.9.9`.

## Migration and data impact
- `2.9.9` adds schema changes to `app_integrations` through migrations:
  - `v54` `log_export_backend`, `log_export_host`, `log_export_port`
  - `v55` last-validation fields for status, message, endpoint, and timestamp
  - `v56` `log_export_host_label`, `log_export_service`
  - `v57` `log_export_debug`
- These migrations are additive and backward-compatible.
- Existing deployments should run the normal migration path before expecting saved External Logs control-plane state to persist.

## Rollback guidance
- Roll back application images to `2.9.8` if the External Logs control-plane workflow needs to be reverted.
- Because `2.9.9` introduces additive schema migrations only, rollback is primarily an image/runtime rollback; older code will ignore the added `app_integrations` fields rather than requiring destructive schema reversal.
