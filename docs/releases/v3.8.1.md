# collectZ v3.8.1

## Version and date

- Version: `3.8.1`
- Date: `2026-05-18`

## Milestone target and status

- Milestone: `3.8.1 — Capture Inbox Photo Upload Intake`
- Status: `implemented`

## Summary

This release turns Capture Inbox photo entries into real backend-owned uploads instead of only storing pasted image paths.

## What changed

- Added authenticated `POST /api/capture-items/upload-image` for multipart photo capture uploads.
- Stored uploaded capture images through the existing backend storage provider and saved the returned image path on the capture row.
- Added web Capture Inbox photo upload support with mobile-friendly image selection.
- Added capture image thumbnails in the inbox list.
- Documented the multipart upload contract in OpenAPI.

## Breaking changes

- None.

## Environment/config changes

- No new environment variables.
- Upload storage follows the existing `STORAGE_PROVIDER` and S3/local upload configuration.

## Migration and data impact

- No new database migration.
- Capture image uploads reuse the `capture_items.image_path` field added in `3.8.0`.

## Security triage

- Photo upload requires existing authentication and scope access.
- Personal access token scope mapping continues to use `media:write` for capture mutations.
- Uploads accept only common image MIME types and use the existing backend upload storage path.

## Security vulnerability triage

- High findings: none identified in local release checks for this slice.
- Owner: collectZ maintainers.
- Target remediation milestone: before `3.8.1` closeout if any high findings appear.

## Deployment and verification

- Verify `/api/health` reports frontend/backend/build `3.8.1`.
- Verify Help > Releases includes `3.8.1`.
- Verify `POST /api/capture-items/upload-image` creates a scoped `photo` capture with an image path.
- Verify Library > Capture Inbox can save a photo capture and show its thumbnail.

## Rollback guidance

- Revert the upload endpoint/UI changes and restore `3.8.0` version metadata and release feed if photo upload intake causes regressions.
- No schema rollback is required for this slice.

## Known issues and follow-up

- OCR and image recognition are not automatic yet.
- Offline queue reconciliation and automatic review resolution remain future `3.8.x` work.
