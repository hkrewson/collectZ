## Summary
This release normalizes signature proof upload and removal around shared `signature_records`. Art and media drawers now use the same per-signature proof workflow while existing object-level proof fields and older compatibility endpoints remain intact.

## What changed
### Shared signature proof workflow
- Added per-signature proof upload and removal endpoints for Art signature records.
- Added matching per-signature proof upload and removal endpoints for media signature records.
- Proof uploads now update `signature_records.proof_path` directly for the selected signature.
- Proof removal clears only that signature record's proof path.

### Drawer experience
- Moved proof file upload/remove controls into the shared signature manager used by Art and media drawers.
- Signature rows can upload, replace, open, and remove proof images from the same UI pattern.
- The archive action now has a clearer accessible label so it does not collide with the proof removal action.

### Compatibility
- Existing Art primary-proof endpoints remain available for older clients.
- Existing media signing-proof endpoints remain available for older clients.
- Primary Art proof readback continues to surface through `signature_proof_path`.
- Primary media proof readback continues to surface through `signed_proof_path`.

### Regression coverage
- Updated OpenAPI with the shared per-signature proof endpoints.
- Added source-level coverage for the shared proof service and route contract.
- Expanded Art and media browser/API regressions to prove per-signature proof upload, primary compatibility projection, and proof removal.

## Breaking changes
- No user-facing breaking changes are expected.
- Existing proof paths remain readable and compatibility endpoints are preserved.

## Environment/config changes
- No new required operator configuration in `3.4.16`.

## Migration and data impact
- No schema migration is included in this release.
- This release continues to use the existing single `signature_records.proof_path` column.
- Multi-proof-per-signature storage remains future work and is not introduced here.
