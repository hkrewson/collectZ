# v2.9.6

## Version and date
- Version: `2.9.6`
- Date: `2026-04-04`

## Milestone target and status
- Target milestone: `2.9.6 — Product Edition Boundary and Homelab Surface Definition`
- Status: `Completed`

## Summary
This release introduces the first explicit `platform` versus `homelab` product-edition boundary inside the private repo. The backend now owns the active edition contract, the frontend shell and Help surface follow that server-owned shape, and platform-only control-plane routes no longer leak into homelab. The release also adds repeatable runtime smoke gates and CI enforcement so both sides of the split stay proven as the repo moves toward a later product/repo separation.

## What Changed
### Product edition boundary
- Added a backend-owned edition contract with supported values:
  - `platform`
  - `homelab`
- Exposed the active `product_edition` through auth bootstrap so the frontend reads server-owned edition state instead of client-only assumptions.
- Added a frontend shell composition boundary for homelab so platform control-plane surfaces are excluded as mounted routes and nav affordances, not just hidden late in rendering.

### Homelab shell and Help surface
- Homelab Help now mounts a shared Help experience with:
  - `Guidance`
  - `Releases`
- Homelab does not mount:
  - `Metrics`
  - `Support`
  - `Help Admin`
- Homelab shell/nav no longer exposes platform control-plane affordances such as:
  - `All Spaces`
  - `All Members`
  - `Activity`
  - `My Space`
- Direct homelab navigation to platform-only dashboard tabs is redirected to homelab-safe destinations.

### Backend route-group separation
- Split backend route mounting into shared versus platform-only groups so homelab no longer mounts platform-only APIs such as:
  - support request and staff control-plane endpoints
  - docs
  - metrics
  - spaces
  - admin spaces
  - admin users
  - explicit auth support-session controls
- Kept shared homelab-safe surfaces mounted, including:
  - auth bootstrap
  - profile
  - libraries
  - general settings
  - integrations
  - feature flags
  - `Help > Releases`

### Homelab product-shape tightening
- Homelab shared auth/scope/library responses now keep:
  - valid `active_library_id`
  - library lists
- While no longer surfacing:
  - `active_space_id`
  - `spaces`
- Homelab registration is now invite-free after the first admin, removing the remaining dependency on the platform tenant-invite model for local account creation.
- Homelab rejects generic authenticated space switching through shared scope APIs.
- Homelab still allows meaningful library creation and library selection inside the single-context product shape.

### Runtime proof and CI enforcement
- Added a repeatable homelab edition-boundary smoke gate to verify the shared mounted surfaces and unmounted platform-only APIs against a live homelab stack.
- Added a matching platform edition-boundary smoke gate to verify that invite-based registration and the tenant/admin control plane remain mounted in platform.
- Wired both edition gates into the blocking CI/release workflow so publish and preflight paths now depend on both sides of the product split.

## Breaking Changes
- Homelab builds no longer expose platform-only control-plane pages or APIs.
- Homelab no longer supports generic authenticated space selection through shared scope APIs.
- Homelab registration behavior now differs intentionally from platform:
  - platform still requires invites after the first user
  - homelab allows additional local accounts without tenant invites

## Environment and Config Changes
- Added/standardized `APP_EDITION` as the backend-owned runtime selector for:
  - `platform`
  - `homelab`
- Existing deterministic browser/smoke verification continues to rely on `PLAYWRIGHT_E2E_BYPASS_TOKEN` in local and CI verification environments.

## Migration and Data Impact
- No schema fork is introduced in `2.9.6`.
- Homelab still operates internally on a single default/personal space for compatibility, but that space model is now treated as implementation detail rather than surfaced client product language.
