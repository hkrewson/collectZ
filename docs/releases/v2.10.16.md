## Summary
This patch release turns the SaaS auth surface into a real public account-entry flow. Users can now request password resets from the sign-in screen, self-register when the platform allows it, verify their email before normal sign-in, and land in the correct workspace model depending on how they were onboarded. Public self-registration now creates a personal workspace after verification, while invite-based onboarding still lands a new user only in the invited workspace without creating an extra personal one.

## What changed
### Password recovery from the auth surface
- Added a public `Forgot password?` flow to the sign-in surface.
- Added a non-enumerating password-reset request endpoint that issues one-time reset tokens and sends reset mail through the platform-owned SMTP path introduced in `2.10.15`.
- Kept reset consumption as a one-time link flow and aligned it with the new public recovery contract in the OpenAPI surface.

### Platform-controlled self-registration
- Added a truthful public auth configuration contract so the sign-in screen knows whether public registration is actually available.
- Added a platform-owned `self_registration_enabled` feature flag so operators can open or close self-registration without misleading auth-screen affordances.
- Preserved invite-required registration behavior when public self-registration is disabled for a non-homelab SaaS deployment that already has users.

### Email verification before sign-in
- Added one-time email-verification tokens and verification-delivery mail for public SaaS self-registration.
- New public registrations now remain pending until email verification succeeds.
- Blocked unverified accounts from normal sign-in and added resend-verification handling on the auth surface.
- Kept the response contract non-destructive when mail delivery fails by preserving the account and allowing verification resend from the sign-in screen.

### Correct workspace landing model
- Public self-registration now creates a dedicated personal workspace and makes that workspace the user’s initial active scope after verification.
- Invite-based onboarding remains separate:
  - the invited user is immediately verified,
  - lands only in the invited workspace,
  - and does not receive an unintended personal workspace.
- Fixed an admin control-plane regression so `All Workspaces -> Workspace Controls -> Add People` can generate invite links again.

### API and product-surface truthfulness
- Synced the auth, verification, and password-reset surfaces into OpenAPI.
- Kept platform settings responsible for self-registration availability while leaving workspace invite/member lifecycle expansion for the next milestone.
- Preserved the distinction between:
  - public SaaS self-registration
  - invite-based workspace onboarding

## Breaking changes
- No backend or API-breaking changes are introduced in `2.10.16`.
- The behavioral contract is stricter for public SaaS sign-in because unverified self-registered accounts must now verify email before normal login succeeds.

## Environment/config changes
- No new required environment variables are introduced in `2.10.16`.
- Public self-registration depends on the platform SMTP foundation from `2.10.15` so verification delivery can function truthfully.
- Platform operators should verify:
  - SMTP remains configured and test-send works
  - `self_registration_enabled` reflects the intended registration policy

## Migration and data impact
- `2.10.16` adds verification state and token persistence for public SaaS signup:
  - `users.email_verified`
  - `users.email_verified_at`
  - `email_verification_tokens`
- Existing users are migrated into a verified state by the release migration path so the new verification gate does not strand established accounts.
- New public self-registrants now receive a personal workspace; invite-claimed users do not.
