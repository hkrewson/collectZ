## Summary
This patch release hardens post-merge duplicate avoidance across changing provider contracts and degraded sync payloads. It proves that a canonical created by one source family can be reused by a later Plex sync through TMDB identity, and that a metadata-poor follow-up payload can still resolve through preserved merge aliases instead of recreating a duplicate. The release also fixes a real metadata-clobbering bug exposed by the new sparse alias proof.

## What Changed
### Cross-provider canonical reuse
- Added runtime proof that a movie canonical first created by a non-Plex source is reused by a later Plex sync through `tmdb_id`.
- Proved the reused row keeps its seeded title variant, showing the update happened through strong identity rather than weak title fallback.

### Sparse alias reuse and metadata preservation
- Added runtime proof that a metadata-poor follow-up CSV payload still resolves to the canonical row through a preserved `provider_item_id` alias after merge.
- Fixed imported media updates so sparse `type_details` merge into the existing canonical metadata instead of replacing richer fields wholesale.
- Proved richer canonical metadata such as author and ISBN survive the degraded follow-up payload.
