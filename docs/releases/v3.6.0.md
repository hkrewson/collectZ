# collectZ v3.6.0

## Version and date

- Version: `3.6.0`
- Date: `2026-05-17`

## Milestone target and status

- Milestone: `3.6.0 — Wishlist and Acquisition Foundation`
- Status: `implemented`

## Summary

This minor milestone starts first-class wishlist and acquisition tracking: a way to track things users want without pretending they already own them.

## What changed

- Promoted `Acquisition, Wishlist, and Want Tracking` from the product backlog into the `3.6.0` roadmap.
- Added scoped `wanted_items` storage with `wanted`, `watching`, `preordered`, `ordered`, `acquired`, and `dismissed` states.
- Added authenticated `/api/wishlist` endpoints to list, create, edit, delete, and convert media-compatible wishlist items into canonical library media rows.
- Added a Library > Wishlist surface with status tabs, type/search filters, lightweight acquisition metadata, and an `Add to library` conversion action.
- Documented the wishlist API in OpenAPI and added browser regression coverage for listing and converting a wanted item.
- Moved the working app/version metadata to `3.6.0` so the minor milestone is aligned across app meta, packages, public compose defaults, release notes, and the in-app release feed.

## Breaking changes

- None.

## Environment/config changes

- No new environment variables.

## Migration and data impact

- Adds migration `103`, which creates scoped `wanted_items` storage and indexes for library/status, space/status, provider keys, and linked media rows.
- `init.sql` was updated with matching table/index/trigger/init migration parity for fresh installs.

## Security triage

- Wishlist records are authenticated and active-scope filtered through the existing workspace/library scope access middleware.
- Provider candidate metadata is stored as non-secret JSON context; integration credentials remain out of wishlist payloads.

## Security vulnerability triage

- High findings: none identified in local release checks for this slice.
- Owner: collectZ maintainers.
- Target remediation milestone: before `3.6.0` closeout if any high findings appear.

## Deployment and verification

## Rollback guidance

- Revert migration `103` and the wishlist route/UI wiring, then restore `3.5.1` version metadata and release feed if the minor release must be backed out before deployment.

## Known issues and follow-up

- Provider-specific wishlist intake remains provider-neutral in this slice; Plex/scanner/search/Kavita candidate import flows can now target the wishlist API in later slices.
- Price watch automation, missing-run detection, purchase workflow details, and richer acquisition planning remain deferred until the foundation object and conversion flow are stable.
