## Summary
This release makes the public deployment path homelab-first. The repository now exposes one generated root `docker-compose.yml` for public homelab deployments, defaults the backend runtime to homelab when no private edition is explicitly configured, and moves private platform stack behavior behind untracked local/CI overrides.

## What changed
### Public compose surface
- Replaced multiple tracked root compose variants with one generated public `docker-compose.yml`.
- Added `scripts/generate-public-compose.js` so the public compose can be regenerated from app metadata.
- Removed the tracked registry and homelab overlay compose files.
- The public compose uses prebuilt GHCR backend/frontend images and contains no public app-edition setting.

### Runtime defaults
- Backend product-edition normalization now defaults to homelab for missing or invalid values.
- Explicit private platform behavior remains available when a private/local/CI stack sets the platform runtime override.
- Public env examples no longer ask homelab deployers to choose an app edition.

### Public docs and validation
- Updated README, setup helper output, and Docker deploy docs to use only the single public compose path.
- Added `scripts/validate-public-export-surface.js` to fail when extra tracked root compose files or old public deploy instructions reappear.
- Updated CI/source verification to generate temporary compose overrides for source builds and private platform boundary checks without committing those private compose files.

## Breaking changes
- Public deploy instructions now use the generated root `docker-compose.yml` only.
- Deployments that previously referenced `docker-compose.registry.yml` should switch to:
  - `docker compose --env-file .env pull`
  - `docker compose --env-file .env up -d`
- Private platform/local deployments should use untracked local compose overrides.

## Environment/config changes
- Public homelab deployments no longer need an app-edition env setting.
- `IMAGE_TAG` remains optional and still overrides the image tag used by the public compose.
- `SESSION_SECRET`, `DB_PASSWORD`, and `INTEGRATION_ENCRYPTION_KEY` remain required for production operation.

## Migration and data impact
- No database migration is introduced by this release.
- Existing data volumes are preserved if compose project names and named volumes are kept consistent.

## Rollback guidance
- If public compose generation causes deployment issues, restore the previous compose file set from the prior release tag and redeploy the previous image tag.
- Data volumes do not need rollback for this release because there is no schema change.
- Private platform operators should keep local compose overrides outside git and reapply them after application rollback if needed.
