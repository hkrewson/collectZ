# collectZ v3.8.4

## Version and date

- Version: `3.8.4`
- Date: `2026-05-18`

## Milestone target and status

- Milestone: `3.8.4 — Capture Replay Conflict Review`
- Status: `implemented`

## Summary

This release makes idempotent Capture Inbox retries more reviewable by preserving conflicting replay data instead of silently overwriting the existing capture row.

## What changed

- Detects differing title, capture type, object type, barcode, symbology, OCR text, and image path values when a queued capture retry reuses an existing `client_capture_id`.
- Keeps the existing capture values when a replay contains conflicting identifying data.
- Stores conflicting incoming values in `capture_items.review_decision.capture_replay_conflicts`.
- Returns `idempotency.status`, `idempotency.conflict_count`, and `replay_conflicts` in retry responses.
- Documents replay conflict readback in OpenAPI.

## Breaking changes

- None.

## Environment/config changes

- No new environment variables.

## Migration and data impact

- No new database migration.
- Replay conflicts are stored in existing `capture_items.review_decision` metadata.

## Security triage

- Capture create and upload routes remain authenticated and scope-enforced.
- Replay conflict detection runs only after the existing scoped `client_capture_id` lookup.
- Conflicting replay data is preserved as review metadata rather than being applied automatically.

## Security vulnerability triage

- High findings: none identified in local release checks for this slice.
- Owner: collectZ maintainers.
- Target remediation milestone: before `3.8.4` closeout if any high findings appear.

## Deployment and verification

- Verify `/api/health` reports frontend/backend/build `3.8.4`.
- Verify Help > Releases includes `3.8.4`.
- Verify a matching capture retry returns `idempotency.status: matched`.
- Verify a conflicting capture retry returns `idempotency.status: needs_review`, preserves the existing row value, and returns `replay_conflicts`.

## Rollback guidance

- Revert the replay conflict detection changes and restore `3.8.3` version metadata and release feed if replay review metadata causes regressions.
- No schema rollback is required for this slice.

## Known issues and follow-up

- This slice stores conflict evidence but does not add a dedicated conflict resolution UI.
- A later review-queue slice can surface replay conflicts alongside scanner, import, and enrichment review decisions.
