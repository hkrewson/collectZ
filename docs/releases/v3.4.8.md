## Summary
This release gives Art and Collectibles a shared `Fandom / Franchise` metadata field so source, fandom, franchise, and universe context can live beside the item without becoming object taxonomy. Art medium/type and Collectibles category stay focused on what the object is, while fandom context becomes searchable metadata across both libraries.

## What changed
### Shared fandom metadata
- Added a shared freeform `franchise` field to native Art and Collectibles.
- Surfaced the field as `Fandom / Franchise` in Art and Collectibles add/edit/detail/card surfaces.
- Included franchise values in Art and Collectibles search so Event-side purchased item linking can find items by source context.

### Taxonomy boundary
- Kept Collectibles categories focused on object shape, such as Card, Lego, Funko, Toys, Clothing, and similar classes.
- Kept Art medium/type focused on physical or artistic form, such as Original, Print, Comic Panel, Sketch, Commission, and Other.
- Did not rename Collectibles to Fandom.

### Contract and regression coverage
- Added migration `77` for nullable `franchise` columns and indexes on `collectibles` and `art_items`.
- Updated `init.sql`, OpenAPI schemas, validation, and serialization so bootstrap and migrated installs stay aligned.
- Extended unit and browser coverage for shared franchise metadata and Event-side purchased item search readback.

## Breaking changes
- No user-facing breaking changes are expected.
- Existing Art and Collectibles records remain readable with `franchise = NULL`.

## Environment/config changes
- No new required operator configuration in `3.4.8`.

## Migration and data impact
- Adds nullable `franchise` columns to `collectibles` and `art_items`.
- Adds indexes for franchise search on both tables.
- Existing records are not backfilled or reclassified.
- Rollback to an older image may ignore the new columns; keep database backups before downgrading across schema migrations.

- Prefer rolling forward with a corrected `3.4.x` build if franchise metadata behavior needs adjustment.
- If rolling back to `3.4.7`, the app should ignore the new nullable columns, but any franchise values entered under `3.4.8` will not be visible until a build with the field is restored.
