## Summary
This release upgrades the library from a single-format ownership model to a true multi-format ownership model. A single title can now carry more than one owned format without fragmenting into duplicate library records, and the main library add/edit flows, import flows, and format filtering all understand that ownership shape. Backward compatibility is preserved through a derived primary `format` value so the existing library surfaces can keep functioning while `owned_formats` becomes the canonical storage model.

## What changed
### Multi-format ownership model
- Added canonical `owned_formats` support for:
  - books
  - comic books
  - games
  - movies
  - TV
  - audio
- Persisted `owned_formats` on media rows while keeping legacy `media.format` as a derived compatibility/display field.
- Kept movie and TV support for `Digital` ownership so existing libraries do not lose valid format state during the transition.

### Library add/edit flows
- Replaced single-format editing with multi-select format controls in the main library add/edit drawers.
- Refined the format control into a denser, single-row picker so larger format sets stay readable without reverting to a wrapped cloud of pills.
- Preserved compatibility expectations by continuing to derive the primary display format from the selected ownership set.

### Import compatibility
- Extended generic CSV import to accept explicit `owned_formats`.
- Updated import template/examples and import helper text to document multi-format values.
- Updated import upserts so repeated imports merge incoming and existing formats on the same title instead of overwriting ownership state or creating format-only duplicates.
- Updated barcode-based add/import coverage to persist `owned_formats` alongside the compatibility `format`.

### Filtering and compatibility
- Changed the library `Format` filter so it now matches any owned format on a title rather than only the derived primary display format.
- Preserved compatibility for existing views by continuing to surface the derived `format` label in list/card UI while `owned_formats` remains the source of truth underneath.

### Regression coverage
- Added targeted browser coverage for:
  - multi-format create
  - multi-format edit
  - barcode import
  - CSV import
- Added focused source/unit regression coverage so import parsing, owned-format filtering, and browser-spec presence are all guarded in the repo test harness.
- Release closeout also fixed a real support-session browser regression in the scoped tenant library-switch path so the full browser lane stays green while the new multi-format coverage ships.

## Breaking changes
- No auth/API-breaking change is introduced for existing clients that still read the derived `format` field.
- Library filtering behavior is intentionally improved:
  - `Format` now matches any owned format a title has
  - it no longer means only the derived primary display format

## Environment/config changes
- No new runtime environment variables are required for `2.10.0`.
- Existing deployments do not need env changes to use the multi-format ownership model.

## Migration and data impact
- `2.10.0` adds schema migration `v58` for multi-format ownership persistence.
- `owned_formats` becomes the canonical stored ownership field.
- The legacy `format` field remains present as a derived compatibility value.
- Existing rows continue to work because compatibility values are preserved and normalized through the release path.
- Repeated imports of the same title now merge owned formats instead of fragmenting records solely by format differences.

## Rollback guidance
- Roll back application images to `2.9.9` if the multi-format ownership cutover needs to be reverted.
- Because `2.10.0` uses additive schema migration and a compatibility `format` field, rollback is primarily an image/runtime rollback. Older code will not understand the new ownership model fully, but the schema change does not require destructive reversal as part of immediate rollback.
