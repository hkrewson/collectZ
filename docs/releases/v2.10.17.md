## Summary
This patch release finishes the workspace-owned member lifecycle foundation. Workspace owners and admins can now force password resets, suspend and restore members within the workspace boundary, invite a brand-new person to become the initial owner during workspace creation, and remove a member from a workspace without silently deleting the content they contributed there. The release keeps platform-global administration distinct from workspace-scoped lifecycle control and preserves the intended tenancy model.

## What changed
### Workspace-scoped password reset
- Added workspace-owned member password reset actions in `Workspace > People`.
- Workspace owners/admins can:
  - email a password reset link to an allowed member,
  - or generate a copyable reset link when explicit link handling is needed.
- Role boundaries are enforced:
  - owners can reset admins/members/viewers,
  - admins can reset members/viewers,
  - normal workspace lifecycle actions do not reset owners through tenant-scoped controls.

### Invited-owner onboarding during workspace creation
- `All Workspaces -> Create Workspace` can now designate a brand-new invited owner instead of requiring an already-active account in the initial owner dropdown.
- The invited owner becomes the first real owner at claim time rather than the current platform admin being silently assigned ownership as a fallback.
- The create-workspace UI now makes the invited-owner state explicit when one onboarding invite is marked `owner`.

### Workspace-scoped suspension
- Added reversible member suspension at the workspace level.
- Suspended members lose access to that workspace and its libraries until restored.
- If the suspended workspace was currently active for that user, their active scope is cleared instead of falling back into access they should no longer keep.
- Suspension remains auditable through workspace activity rather than acting like a hidden platform-global account lock.

### Safe workspace removal semantics
- Clarified workspace member removal as `Remove` within the workspace People context rather than ambiguous user deletion wording.
- Removing a member revokes workspace access but preserves shared workspace content.
- Last-owner protection remains enforced:
  - a workspace cannot lose its final owner through normal member management.
- Library-ownership transfer rules still block removal when explicit ownership transfer is required first.

### Roadmap clarity for later offboarding work
- The bigger personal-workspace account deletion/archive/recovery policy is now tracked separately in a later milestone instead of being mixed into workspace membership removal.
- The roadmap now captures the intended personal-workspace retention lifecycle:
  - `0-30 days`: reclaim directly on same-email re-registration
  - `day 31`: move data from live into archive/backup pool
  - `day 91`: permanently delete archived data

## Breaking changes
- No API-breaking changes are introduced in `2.10.17`.
- The behavioral contract is stricter in a good way:
  - workspace suspension now actively removes access instead of leaving scope/library access ambiguous,
  - workspace removal is explicitly non-destructive for contributed content.

## Environment/config changes
- No new required environment variables are introduced in `2.10.17`.
- The milestone continues to rely on the platform SMTP foundation from `2.10.15` for invite and reset email delivery.

## Migration and data impact
- `2.10.17` adds workspace membership suspension fields:
  - `space_memberships.suspended_at`
  - `space_memberships.suspended_by`
- Existing memberships remain active after migration until explicitly suspended.
- No destructive data migration is part of this release.
