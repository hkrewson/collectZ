# collectZ v3.8.2

## Version and date

- Version: `3.8.2`
- Date: `2026-05-18`

## Milestone target and status

- Milestone: `3.8.2 — Capture OCR Identifier Candidate Review`
- Status: `implemented`

## Summary

This release makes OCR text in Capture Inbox reviewable by extracting likely identifiers and letting the user explicitly choose the right one.

## What changed

- Added authenticated `POST /api/capture-items/{id}/ocr-text` for storing OCR text and extracting ISBN, UPC/EAN, and ASIN candidates.
- Added authenticated `POST /api/capture-items/{id}/apply-ocr-candidate` for applying a selected OCR candidate to a capture.
- Stored OCR candidate lists and selected candidate metadata in capture review data.
- Added compact OCR candidate controls in Library > Capture Inbox.
- Documented the OCR text and candidate apply contracts in OpenAPI.

## Breaking changes

- None.

## Environment/config changes

- No new environment variables.
- This slice does not add a backend OCR provider or image-recognition dependency.

## Migration and data impact

- No new database migration.
- OCR candidates are stored in existing `capture_items.review_decision` metadata.

## Security triage

- OCR candidate extraction and candidate application require existing authentication and scope access.
- Personal access token scope mapping continues to use `media:write` for capture mutations.
- Candidates are review metadata only until explicitly applied by the user.

## Security vulnerability triage

- High findings: none identified in local release checks for this slice.
- Owner: collectZ maintainers.
- Target remediation milestone: before `3.8.2` closeout if any high findings appear.

## Deployment and verification

- Verify `/api/health` reports frontend/backend/build `3.8.2`.
- Verify Help > Releases includes `3.8.2`.

## Rollback guidance

- Revert the OCR candidate endpoint/UI changes and restore `3.8.1` version metadata and release feed if OCR candidate review causes regressions.

## Known issues and follow-up

- Backend-owned OCR image processing is not implemented yet.
- Cover/art recognition, offline queue reconciliation, and automatic review resolution remain future `3.8.x` work.
