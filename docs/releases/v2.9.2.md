## Version and date
- Version: `2.9.2`

## Milestone target and status
- Target milestone: `2.9.2 — Explicit Support Request, Consent, and Session Approval`
- Status: `Completed`

## Summary
This release finishes the support workflow move from a loosely administrative action into a request-driven, explicitly approved support model. It ships the support foundations introduced during `2.9.1`, including the in-app Help center, Help Admin, release-note visibility, and support triage, then completes the `2.9.2` consent and session work by linking support sessions to approved requests, limiting `support_admin` tenant access to approved request context, and making the support UI easier to operate with clearer request identity, chat-style history, and compact operational controls.

## What Changed
### Help center and Help Admin foundations
- Added the tabbed in-app Help experience with:
  - `Guidance`
  - `Releases`
  - `Support`
- Added a dedicated Help Admin staff surface that unifies support reply and triage work instead of splitting it across separate admin views.
- Added lightweight in-app release notes so users can see recent shipped changes without leaving the product.
- Added queue badges and clearer support request identity with visible `SUP-######` request keys.

### Support requests, triage, and user-visible status
- Added first-class support request creation and threaded replies.
- Added staff triage controls for:
  - request classification (`support`, `bug`, `feature request`)
  - tracking status
  - linked repo issue fields for later operational use
  - resolved/shipped version metadata
- Added staff-only internal notes that save inline with the thread while remaining hidden from the requester.
- Added support-side close and requester-side reopen actions so case lifecycle does not depend on one side remembering to resolve it.

### Consent and request-linked support access
- Added explicit requester approval and revocation controls for support access on support requests.
- Support access approval now creates clear in-thread system history instead of relying on out-of-band assumptions.
- `support_admin` support-session starts are now request-linked and approval-gated:
  - no freehand tenant entry for non-admin support staff
  - the request must be approved
  - the request target must match the support-session target
- Active support sessions now carry request identity so the authorized request stays visible during the session lifecycle.
- `admin` retains the broader break-glass path as a separate operator capability.

### Help Admin operations UX
- Help Admin now supports a denser operator workflow:
  - compact queue summary
  - `Reply | Triage` controls
  - support metrics tab with:
    - time to open
    - time to close
    - closed this month
- Guidance no longer mixes in redundant workload chrome for support staff.
- Queue context now appears above the conversation thread so the thread column holds both operational context and the active case discussion.

## Breaking Changes
- `support_admin` no longer has an arbitrary freehand tenant support-session path.
- Non-admin support staff must work from an approved support request before entering tenant context.
- Help/Admin support operations are now centered in the unified Help surfaces rather than older split mental models.

## Environment and Config Changes
- No new required environment variables were introduced for `2.9.2`.
- The release-note feed consumed by `Help > Releases` must continue to be regenerated during release closeout.

## Migration and Data Impact
- This release adds schema needed for the support workflow:
  - support request tracking and threading foundations
  - staff-only internal note visibility
  - explicit support-access approval state
  - support-session linkage to approved support requests
- Existing user/media data remains valid.
- Support and consent state now persist in dedicated support request and session fields instead of being implied by operator behavior.
