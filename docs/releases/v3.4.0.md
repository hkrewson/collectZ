## Summary
This minor release promotes Art into its own first-class library surface while preserving event-linked purchase history and cleaning up the remaining Collectibles entry friction that no longer made sense once Art stopped living only as a subtype. The release keeps the underlying bridge model intentionally pragmatic: Art now has dedicated routes and navigation, shared collectible-style purchase linkage still works, Art metadata grows into a more natural shape with `series`, `vendor`, and `booth`, and the Collectibles drawer collapses its duplicate taxonomy controls into one clearer selector.

## What changed
### First-class Art library surface
- Added a dedicated `Art` library destination to the main product shell.
- Added Art-specific backend routes:
  - `GET /api/art/categories`
  - `GET /api/art`
  - `POST /api/art`
  - `GET /api/art/:id`
  - `PATCH /api/art/:id`
  - `DELETE /api/art/:id`
  - image upload and removal endpoints
- Kept Art scoped to Art-only records at the route boundary instead of treating the tab as only a frontend filter.

### Event purchase linkage preserved
- Kept Art linked to Events through the existing shared collectible-style relationship path during the bridge milestone.
- Preserved current event-linked purchase behavior instead of introducing a brand-new detached purchase ledger prematurely.
- Carried forward the roadmap contract for the next `3.4.x` line:
  - freeform event artifacts remain the note/history lane
  - tracked purchases will move toward a shared purchased-item relationship for Art and Collectibles

### Art-native metadata improvements
- Added `Series` as a first-class field for Art and reusable collectible-style entries like comic panels.
- Split the overloaded `Vendor/Booth` field into distinct:
  - `vendor`
  - `booth`
- Kept backward-compatible read behavior for older combined values while aligning the active editor/detail contract with the split fields.

### Collectibles taxonomy simplification
- Removed the redundant two-menu Collectibles editor flow that forced users to choose both `Type` and `Category`.
- Collapsed Collectibles entry to one classification selector.
- Folded `Card` into that same surviving selector.
- Removed the practical need to expose `Collectible` as a separate visible type choice in the Collectibles drawer.

### Contract and release sync
- Updated OpenAPI for the promoted Art surface and expanded metadata shape.
- Updated source-backed regression coverage around:
  - Art route promotion
  - Collectibles taxonomy simplification
  - Art metadata fields
- Regenerated the in-app release feed for `Help > Releases`.
- Synced `3.4.0` version metadata across root, backend, and frontend package/app metadata files.

## Breaking changes
- None intended for end users.
- This is a minor version because it introduces a new first-class library surface and sets up the next native Art separation line, but it does not yet cut Art over to a fully separate storage model.

## Environment/config changes
- No new required operator configuration in `3.4.0`.
- Existing Events and Collectibles feature-flag/runtime controls continue to govern the bridge-phase Art surface.

## Migration and data impact
- `3.4.0` includes the ongoing schema bridge for Art metadata expansion:
  - `series`
  - `vendor`
  - `booth`
- Existing art-classified collectible rows remain on the bridge storage model in this release.
- Full native Art storage migration and shared purchased-item backfill are intentionally planned for the `3.4.1` through `3.4.3` sequence rather than being silently mixed into this milestone.
