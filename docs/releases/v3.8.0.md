# collectZ v3.8.0

## Version and date

- Version: `3.8.0`
- Date: `2026-05-18`

## Milestone target and status

- Milestone: `3.8.0 — Mobile Capture Inbox Foundation`
- Status: `implemented`

## Summary

This release starts the mobile-first capture workflow with a backend-owned Capture Inbox for quick real-world captures.

## What changed

- Added scoped `capture_items` storage for barcode, photo, OCR text, and manual-note captures.
- Added authenticated `/api/capture-items` endpoints so scanner/mobile clients can create and read captures directly.
- Added review actions to update, discard, delete, and convert captures into Wishlist candidates.
- Added Library > Capture Inbox in the web app for reviewing captured items.
- Documented the capture API contract in OpenAPI.

## Breaking changes

- None.

## Environment/config changes

- No new environment variables.

## Migration and data impact

- Adds migration `104`, which creates `capture_items` plus scoped status/barcode indexes.
- Existing barcode lookup/import behavior is unchanged.

## Security triage

- Capture lookup and review routes require existing authentication and scope access.
- Personal access token scope mapping follows the media contract: read endpoints require `media:read`, mutating endpoints require `media:write`.
- Capture records are scoped to the active workspace/library.

## Security vulnerability triage

- High findings: none identified in local release checks for this slice.
- Owner: collectZ maintainers.
- Target remediation milestone: before `3.8.0` closeout if any high findings appear.

## Deployment and verification

- Verify `/api/health` reports frontend/backend/build `3.8.0`.
- Verify Help > Releases includes `3.8.0`.
- Verify `POST /api/capture-items` creates a scoped capture.
- Verify Library > Capture Inbox shows the capture and can convert it to Wishlist.

## Rollback guidance

- Revert the capture inbox route/UI changes and restore `3.7.5` version metadata and release feed if capture intake causes regressions.
- Migration `104` only adds new capture storage; existing library data is not rewritten.

## Known issues and follow-up

- Photo capture currently stores image references, not binary upload handling.
- OCR, cover/art recognition, offline queue reconciliation, and automatic review resolution remain future `3.8.x` work.
