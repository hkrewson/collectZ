## Summary
This release lets Art and media records carry multiple shared signature provenance records while preserving the existing single-primary compatibility fields used by older forms, imports, and title endpoints.

## What changed
### Shared signature records
- Added shared create, update, archive, and set-primary operations for `signature_records`.
- Art and media can now keep multiple active signature records without replacing the existing primary signature.
- Archiving a primary signature promotes the newest remaining active signature so signed-object readback stays coherent.

### Art and media APIs
- Added Art signature endpoints for listing, adding, editing, archiving, and promoting signature records.
- Added matching media signature endpoints with existing ownership/scope enforcement.
- Promoting a signature updates the compatibility projection: Art `signer_*` readback and media `signed_by`, `signed_role`, `signed_on`, `signed_at`, and `signed_proof_path` follow the active primary signature.
- Clearing the legacy single-signature shortcut archives active signatures for that owner, preserving the older “this item is not signed” behavior.

### Readback and UI
- Art and media detail responses can now include more than one active signature in the normalized `signatures` array.
- Art and media detail drawers show an `All signatures` section when multiple signatures exist.
- The existing single-signature forms remain primary-signature shortcuts for compatibility.

### Regression coverage
- Added source-level coverage for the shared multi-signature service, Art/media routes, validation schemas, OpenAPI paths, and UI readback.
- Added browser/API regressions for Art and media that create a secondary signature, promote it to primary, verify legacy fields follow it, then archive it and verify fallback behavior.

## Breaking changes
- No user-facing breaking changes are expected.
- Existing Art, media, event autograph links, and one-signature workflows remain compatible.

## Environment/config changes
- No new required operator configuration in `3.4.13`.

## Migration and data impact
- No schema migration is included in this release.
- Uses the multi-row `signature_records` table and active-primary partial index introduced by earlier signature provenance releases.
- Existing signature rows are not destructively rewritten during deployment.
