## Summary
This release connects event-captured autographs to the shared signature provenance model introduced for Art and media. Events remain the capture/history surface, while Art and media records remain the canonical signed-object evidence surface.

## What changed
### Event autograph provenance
- Added `event_artifact` as a supported owner type in shared `signature_records`.
- Autograph event artifacts can now capture signer name, role, signed date, signed location, proof path, and notes as shared signature provenance.
- Event artifact readback now exposes the event-captured autograph signature and any linked object signature separately.

### Object signature linking
- Added `POST /api/events/:id/artifacts/:artifactId/link-signature` for linking an autograph artifact to an Art or media signature record.
- Linking an autograph to Art creates or updates the Art primary shared signature and marks the Art item signed.
- Linking an autograph to media creates or updates the media primary shared signature and keeps legacy media signature fields compatible.
- The linked object signature preserves the source event through `signed_event_id`.

### API contract and regression coverage
- Updated OpenAPI for Event artifact signature readback and autograph-to-object signature linking.
- Added a Docker runtime smoke for the full Event autograph to Art signature linking workflow.
- Added source-level regression coverage for migration, validation, route, OpenAPI, and smoke-script wiring.

## Breaking changes
- No user-facing breaking changes are expected.
- Existing event artifacts remain readable.
- Existing Art and media signature records remain compatible.

## Environment/config changes
- No new required operator configuration in `3.4.11`.

## Migration and data impact
- Adds migration `79`: `Link event autograph artifacts to shared signature provenance`.
- Adds nullable `event_artifacts.signature_record_id` for links from event autograph artifacts to canonical Art/media signature records.
- Expands the `signature_records.owner_type` constraint to include `event_artifact`.
- Existing rows are not destructively rewritten.

## Rollback guidance
- Prefer rolling forward with a corrected `3.4.x` build if event autograph linking needs adjustment.
- If rolling back application images to `3.4.10`, the additive `signature_record_id` column and expanded owner-type constraint may remain; older code ignores the event artifact signature link field.
- Avoid using newly linked event autograph provenance until `3.4.11` or later is restored if the app image is rolled back.

## Known issues and follow-up
- This release exposes the linking contract through the API, but does not add a dedicated frontend linking workflow yet.
- Events are not signed objects themselves; signings remain event artifacts/sub-events.
- Multiple signatures and multi-proof management remain future work.
