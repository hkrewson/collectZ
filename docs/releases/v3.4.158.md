# collectZ v3.4.158

## Version and date

- Version: `3.4.158`
- Date: `2026-05-16`

## Milestone target and status

- Milestone: `3.4.158 — Barcode Scanner Multi-Candidate Lookup`
- Status: `released`

## Summary

This patch keeps scanner barcode lookup as a candidate search by preserving multiple provider products for a single barcode instead of allowing duplicate provider identifiers to collapse choices in scanner clients.

## What changed

- Preserved multiple barcode provider matches in `POST /api/media/lookup/barcode`.
- Added stable per-candidate provider IDs so barcode variants sharing the same UPC stay distinct.
- Added `provider_candidate_index` to scanner lookup matches when provider ordering is available.
- Updated scanner smoke coverage with a fake UPC provider returning three candidates for `0076783005990`.
- Clarified OpenAPI docs that barcode lookup may return zero, one, or many candidates.

## Breaking changes

- None.

## Environment/config changes

- None.

## Migration and data impact

- No database migration.
- Existing scanner imports keep using the selected match supplied by the client.

## Security triage

- Lookup remains authenticated with `media:read` because catalog match readback can expose private collection data.
- Import auth behavior is unchanged and remains token-gated with `import:run`.
- The scanner smoke uses a local fake barcode provider and does not persist provider tokens or secret-bearing evidence.

## Security vulnerability triage

- High findings: none in local dependency audits.
- Owner: collectZ maintainers.
- Target remediation milestone: not needed for this release because no high findings were present locally.
