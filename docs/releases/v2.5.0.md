# v2.5.0

## Version and date
- Version: `2.5.0`
- Date: `2026-03-13`

## Milestone target and status
- Target milestone: `2.5.0 — Invite/Reset Security and Secret Exfiltration Hardening`
- Status: `Completed with one minor follow-up note`

## Summary
This release hardens invite/reset delivery and token exposure, locks browser auth back to cookie-session defaults, adds proper non-browser API authentication for automation, improves import job behavior, and adds the operator runbooks needed to recover from auth or SMTP issues safely.

## What changed
### Frontend
- Added PAT management to Profile so session-authenticated users can create, copy once, review, and revoke API tokens.
- Updated Admin Members invite/reset flows so copy-link exposure is explicit instead of the default path.

### Backend
- Added SMTP-backed invite and admin-issued password-reset delivery with activity coverage for create, delivery success, delivery failure, token exposure, consume, and invalidation flows.
- Hardened auth so browser bearer-session fallback is disabled by default and mutating browser flows stay on cookie + CSRF.
- Added PAT support with hashed-at-rest storage, scopes, revoke, optional expiry, and route-family scope enforcement.
- Added service-account keys for machine-to-machine access with:
  - admin-only creation/revocation,
  - one-time reveal,
  - hashed-at-rest storage,
  - explicit route-prefix allowlists,
  - no admin wildcard scope.
- Improved async import API behavior:
  - import launch endpoints now queue by default,
  - status endpoints return compact summaries by default,
  - full result payload moved to `/api/media/sync-jobs/:id/result`.
- Fixed Plex/TMDB enrichment issues:
  - TV season enrichment now runs only for true TV rows,
  - movie TMDB ids no longer hit `/tv/:id`,
  - TMDB request diagnostics are captured for result inspection,
  - poster lookup matching now uses broader normalized title candidates.
- Added regression checks for:
  - audit redaction,
  - integration secret masking,
  - PAT auth behavior,
  - service-account route enforcement,
  - CSRF browser-vs-bearer behavior,
  - TMDB/Plex enrichment guardrails.

### Database
- Added migration `v40`: personal access tokens.
- Added migration `v41`: service account keys.
- `init.sql` now mirrors both auth tables and update triggers.

### Docs/Ops
- Added PAT usage guide:
  - `docs/wiki/25-Personal-Access-Tokens.md`
- Added service-account key guide:
  - `docs/wiki/27-Service-Account-Keys.md`
- Expanded recovery/ops runbooks:
  - `docs/wiki/15-Secrets-and-Rotation-Runbook.md`
  - `docs/wiki/26-Admin-Recovery-and-SMTP-Triage.md`
- Updated env/config docs for SMTP and operator recovery references.

## Breaking changes
- Browser `Authorization: Bearer <session>` fallback is disabled by default.
- Non-browser automation should use PATs or service-account keys instead of browser session reuse.

## Environment/config changes
- Added/used:
  - `ALLOW_SESSION_BEARER_FALLBACK`
  - SMTP env vars for invite/reset delivery:
    - `SMTP_HOST`
    - `SMTP_PORT`
    - `SMTP_SECURE`
    - `SMTP_USER`
    - `SMTP_PASS`
    - `SMTP_FROM`

## Migration and data impact
- Additive schema changes only.
- New tables:
  - `personal_access_tokens`
  - `service_account_keys`
- No destructive data migration required.

## Deployment and verification
1. Rebuild and start:
   - `docker compose --env-file .env up -d --build backend frontend`
2. Verify reported version is `2.5.0`.
3. Verify SMTP-backed invite or admin password-reset delivery.
4. Verify PAT flow:
   - create from Profile,
   - call a read route,
   - revoke and confirm token rejection.
5. Verify service-account flow:
   - create via admin session,
   - confirm allowed route works,
   - confirm disallowed prefix/scope fails,
   - revoke and confirm token rejection.
6. Verify browser mutating routes still require CSRF with cookie-session auth.
7. Verify import launch endpoints return queued job responses and compact job polling output.

## Rollback guidance
- Roll back application images if needed.
- DB can remain forward on `v40` and `v41`; both migrations are additive.
- If rolling back to code that does not understand service-account keys, do not use newly-created service-account credentials until the newer app version is restored.
