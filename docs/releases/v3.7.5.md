# collectZ v3.7.5

## Version and date

- Version: `3.7.5`
- Date: `2026-05-18`

## Milestone target and status

- Milestone: `3.7.5 — Timeline Event Payload Enrichment`
- Status: `implemented`

## Summary

This patch enriches newly logged event activity so the human-readable timeline can show event titles and child object names instead of only ids.

## What changed

- Added shared event activity detail builders for event title, date, location, and related child object context.
- Event attendee rows now log attendee name, status, visibility, linked user id, and event context.
- Event group, meetup, schedule plan, and schedule session rows now log their own readable name/title plus event context.
- Schedule preview, notification, and schedule import rows now include event context in their activity details.
- Updated browser coverage to prove new attendee activity shows the attendee name in Workspace Activity.

## Breaking changes

- None.

## Environment/config changes

- No new environment variables.

## Migration and data impact

- No database migration.
- Historical activity rows are not rewritten; enrichment applies to newly logged event activity.

## Security triage

- Activity enrichment uses data already visible through scoped event APIs.
- No new public routes or broader activity permissions were added.

## Security vulnerability triage

- High findings: none identified in local release checks for this slice.
- Owner: collectZ maintainers.
- Target remediation milestone: before `3.7.5` closeout if any high findings appear.

## Deployment and verification

- Verify `/api/health` reports frontend/backend/build `3.7.5`.
- Verify Help > Releases includes `3.7.5`.
- Verify creating an event attendee writes activity details containing `eventTitle` and `attendeeName`.
- Verify Workspace > Activity > Events shows the readable attendee name for new event activity.

## Rollback guidance

- Revert the event activity detail-builder changes and restore `3.7.4` version metadata and release feed if event logging regressions appear.

## Known issues and follow-up

- Existing historical event activity remains as sparse as originally logged.
- Provider writeback histories still use their existing payloads and can be enriched separately.
