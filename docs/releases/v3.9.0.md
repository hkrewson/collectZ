# collectZ v3.9.0

## Version and date

- Version: `3.9.0`
- Date: `2026-05-20`

## Milestone target and status

- Milestone: `3.9.0 — Apple/iTunes Wishlist Search Intake`
- Status: `implemented`

## Summary

This release starts the Apple/iTunes wishlist line with a backend-owned search and save flow for adding Apple Store candidates directly to Wishlist.

## What changed

- Added authenticated `GET /api/wishlist/apple-itunes/search` for Apple/iTunes candidate search.
- Added authenticated `POST /api/wishlist/apple-itunes/save` for scoped Wishlist saves.
- Normalized Apple result shapes across movies, TV, ebooks, audiobooks, music, podcasts, short films, and software.
- Preserved Apple provider identity using `provider='apple_itunes'` and stable provider keys.
- Added scoped duplicate readback so saved Apple-linked results show as saved instead of creating confusing duplicate Wishlist rows.
- Added a compact Apple/iTunes search panel to Library > Wishlist with artwork, price/currency, store link, optional target price, and direct Add action.
- Documented the new Apple/iTunes Wishlist contract in OpenAPI.

## Breaking changes

- None.

## Environment/config changes

- No new required environment variables.
- `APPLE_ITUNES_SEARCH_URL` is available only as a test/runtime override for the backend Apple Search API base URL.
- Apple/iTunes Search API use does not require an API key.

## Migration and data impact

- No database migration.
- New Wishlist rows saved from Apple/iTunes store provider metadata in existing `wanted_items` columns.

## Security triage

- Apple/iTunes search remains authenticated so scoped duplicate state and provider usage stay behind the backend.
- No provider secrets are introduced.
- Apple raw result metadata is stored only inside authenticated/scoped Wishlist rows.

## Security vulnerability triage

- High findings: none identified in local checks for this slice.
- Owner: collectZ maintainers.
- Target remediation milestone: before `3.9.0` closeout if any high findings appear.
