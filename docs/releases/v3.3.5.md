## Summary
This patch release rounds out the current library-loans reminder line by adding an operator-facing reminder-operations surface. The release keeps the scope intentionally narrow: instead of creating a separate monitoring dashboard, it adds a dedicated admin read contract and a compact panel inside the existing platform activity view so operators can see reminder runtime state, latest automatic run results, and recent failures in one place.

## What changed
### Reminder operations backend contract
- Added a dedicated admin endpoint for automatic reminder operations visibility.
- `GET /api/admin/loan-reminder-operations` now returns:
  - runtime state
  - latest automatic reminder run
  - recent automatic run summaries
  - recent automatic reminder failures
- Reused existing reminder automation audit events instead of inventing a separate reporting store.

### Admin operator surface
- Added a compact reminder-operations panel to the existing `Platform Activity` page.
- Surfaced runtime status for automation and SMTP availability.
- Surfaced latest automatic run counts for sent, eligible, skipped, and failed reminders.
- Surfaced recent automatic failure entries with phase, borrower, and timestamp context.
- Kept the generic activity feed intact below the new reminder-operations section.

### Contract and release sync
- Documented the reminder-operations response shape in OpenAPI.
- Updated source-backed regression coverage for the admin route, OpenAPI contract, and admin activity surface.
- Regenerated the in-app release feed for `Help > Releases`.
- Synced `3.3.5` version metadata across root, backend, and frontend package/app metadata files.

## Breaking changes
- None intended for operators or end users.
- `3.3.5` is a patch follow-up on the shipped loans, reminder automation, reminder history, and global loan history work, not a redesign of the loan lifecycle or admin shell.

## Environment/config changes
- No new required operator configuration.
- Existing automatic reminder and SMTP settings continue to govern reminder runtime behavior.

## Migration and data impact
- No new schema migration is introduced in `3.3.5`.
- The release reuses existing activity log data from reminder automation and existing reminder-history persistence from prior milestones.
